# ImageView 空状态设计

日期：2026-07-11

状态：已批准实施

## 1. 目标

ImageView 当前在没有打开图片时只显示黑色画布和无效的底部状态值，用户容易误以为应用未正常工作。本次在不增加图片库、最近文件首页或其他大型功能的前提下，为真正无图片的窗口增加明确、可操作的原生空状态。

## 2. 范围

本次包含：

- 在画布中央显示图片图标、提示文字和“打开图片…”按钮。
- 保留现有 Finder 拖放与 `Command-O` 打开方式。
- 点击空状态按钮时复用现有文件选择和多图片打开流程。
- 首次启动及最后一张图片被移除后显示空状态。
- 加载、成功显示和失败状态不显示空状态。
- 真正无图片时隐藏只对图片有意义的底部尺寸、页码、缩放值和信息按钮。
- 提供简体中文和英文文案、键盘焦点及辅助功能标签。

本次不包含：

- 最近文件列表、收藏、图库管理或欢迎页。
- 示例图片、营销内容、装饰背景或首次使用向导。
- 图片解码、编辑、缓存、目录扫描及文件关联逻辑改造。
- 新的加载动画或错误恢复流程。

## 3. 用户界面

空状态使用独立的原生 AppKit 视图，居中覆盖在图片画布上方，并保持较低视觉权重：

1. 系统图片符号，使用次要文字颜色。
2. 标题。
3. 一行辅助说明。
4. 标准 macOS 按钮。

简体中文文案：

- 标题：`打开图片`
- 说明：`将图片拖到这里，或点击下方按钮`
- 按钮：`打开图片…`

英文文案：

- 标题：`Open an Image`
- 说明：`Drag an image here, or choose one below`
- 按钮：`Open Image…`

视图遵循系统深色、浅色和高对比度设置，不绘制自定义背景卡片、虚线拖放框或大面积装饰。按钮可以通过键盘获得焦点，图标不单独进入辅助功能树；标题、说明和按钮提供明确的辅助功能语义。

## 4. 状态规则

`ImageLoadPhase` 增加 `loading` 阶段，以明确区分“窗口真正为空”和“已开始打开但尚未得到预览图”。空状态可见性的唯一规则为：

```text
currentImage == nil
AND loadPhase == empty
AND errorMessage == nil
```

状态转换：

| 场景 | 阶段 | 空状态 | 错误提示 | 图片状态栏内容 |
| --- | --- | --- | --- | --- |
| 首次启动 | `empty` | 显示 | 隐藏 | 隐藏 |
| 最后一张图片被移除 | `empty` | 显示 | 隐藏 | 隐藏 |
| 开始打开图片 | `loading` | 隐藏 | 隐藏 | 隐藏 |
| 预览图到达 | `preview` | 隐藏 | 隐藏 | 按现有逻辑显示 |
| 完整图到达 | `full` | 隐藏 | 隐藏 | 按现有逻辑显示 |
| 打开失败 | `failed` | 隐藏 | 显示现有错误 | 隐藏 |

打开新图片时，在启动异步解码之前同步进入 `loading`，因此不会在慢速解码期间继续显示“打开图片”入口。现有 generation 隔离及渐进加载竞态规则保持不变。

## 5. 组件与职责

### 5.1 `EmptyStateView`

新增独立 AppKit 视图，负责：

- 组装系统图标、标题、说明和按钮。
- 使用 `AppStrings` 读取中英文文案。
- 通过 `onOpenRequested` 闭包报告按钮点击。
- 暴露最小的测试表面，用于验证文案、辅助功能和回调。

该组件不创建文件选择器，不持有 `ViewerViewModel`，也不负责判断自身是否应该显示。

### 5.2 `MainWindowController`

主窗口负责：

- 将 `EmptyStateView` 添加到画布上层并居中约束。
- 根据 `currentImage`、`loadPhase` 和 `errorMessage` 统一更新空状态可见性。
- 在空状态显示时隐藏底部尺寸、页码、缩放值和信息按钮；图片出现后恢复现有状态栏逻辑。
- 将空状态按钮点击转发到窗口级 `onOpenRequested` 闭包。

可见性判断提取为无副作用的纯函数，避免三个 Combine 订阅各自维护一套条件。

### 5.3 `AppDelegate`

`AppDelegate` 继续拥有系统文件选择器和多窗口打开策略：

- 将现有 `openImage(_:)` 中的面板逻辑提取为可复用方法。
- 创建窗口控制器时设置其 `onOpenRequested`。
- 菜单 `Command-O` 与空状态按钮调用同一方法。
- 文件选择器继续允许多选，选中结果继续交给现有 `openURLs(_:)`。

这样不会形成第二套打开、窗口复用或最近项目记录逻辑。

## 6. 错误处理与边界

- 用户取消文件选择器时不改变空状态。
- 拖入不支持或无法解码的文件时沿用现有错误提示，空状态不与错误叠加。
- 从失败状态再次使用菜单或快捷键打开图片时，进入 `loading` 并清除旧错误的现有行为保持不变。
- 最后一张图片因移到废纸篓或外部删除而消失，且没有错误时，窗口回到空状态。
- 图片正在加载、预览已显示或存在未保存编辑时，空状态按钮不出现。
- 空状态不拦截 RootInteractionView 现有文件拖放事件。

## 7. 测试

测试先于生产实现编写，并覆盖：

1. `EmptyStateView` 提供正确的中文和英文标题、说明及按钮文案。
2. 点击按钮只触发一次 `onOpenRequested`。
3. 初始 `empty + nil image + nil error` 显示空状态并隐藏图片状态栏内容。
4. `loading`、`preview`、`full` 和 `failed` 均隐藏空状态。
5. 即使阶段为 `empty`，存在错误时也不显示空状态。
6. 最后一张图片移除并回到无错误 `empty` 后重新显示空状态。
7. 空状态按钮与菜单打开入口复用同一 AppDelegate 打开回调。
8. 现有拖放、渐进加载、错误覆盖、多窗口和本地化测试继续通过。

最终门禁：

```bash
swift test --disable-sandbox
scripts/build-app.sh
test -x .build/ImageView.app/Contents/MacOS/ImageView
plutil -lint .build/ImageView.app/Contents/Info.plist
```

并手动确认首次启动、点击打开、取消面板、拖入图片、打开失败、移除最后一张图片、深色和浅色外观下的表现。

## 8. 验收标准

1. 用户首次看到窗口时能立即理解需要打开或拖入图片。
2. 点击“打开图片…”与 `Command-O` 使用完全相同的多图片打开流程。
3. 加载和错误状态不会与空状态重叠。
4. 最后一张图片被移除后，窗口恢复可操作的空状态。
5. 无图片时不再显示 `— × — px`、`0 / 0`、无意义缩放值或信息按钮。
6. 图片显示、拖放、浏览、编辑和保存行为没有回归。
7. 中英文、键盘焦点、辅助功能、深浅色外观均可用。
8. 全量测试和 Release App Bundle 构建通过。
